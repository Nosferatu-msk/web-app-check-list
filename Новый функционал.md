# ТРЕБУЕМЫЕ ДОРАБОТКИ WEB-ПРИЛОЖЕНИЯ
## Web-приложение «Цифровой чек-лист инженера по техническому обслуживанию оборудования»

---

## ЧАСТЬ I. РОЛЕВАЯ МОДЕЛЬ СИСТЕМЫ

### 1. ОБЗОР РОЛЕВОЙ МОДЕЛИ

#### 1.1. Иерархия ролей

Система построена по четырехуровневой иерархической модели с четким разделением ответственности и прав доступа:

```
┌─────────────────────────────────────────────────────────────┐
│                    АДМИНИСТРАТОР                            │
│         (Настройка системы, управление справочниками,       │
│          назначение объектов и инженеров ТМ)                │
──────────────────────────┬──────────────────────────────────┘
                           │
                           │ Определяет:
                           │ • Какие объекты доступны ТМ
                           │ • Какие инженеры подчиняются ТМ
                           ▼
┌─────────────────────────────────────────────────────────────┐
│            ТЕРРИТОРИАЛЬНЫЙ МЕНЕДЖЕР (ТМ)                    │
│         (Диспетчер / Контролер региона,                     │
│          планирование работ, контроль качества)             │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           │ Управляет:
                           │ • Визитами своих инженеров
                           │ • Объектами своей территории
                           │ • Формированием и отправкой отчетов
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                       ИНЖЕНЕР                               │
│              (Полевой исполнитель)                          │
│         (Создание визитов, заполнение параметров,           │
│          фотофиксация, отправка отчетов)                    │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           │ Отправляет отчеты
                           ▼
┌─────────────────────────────────────────────────────────────┐
│                      ЗАКАЗЧИК                               │
│            (Получатель отчетов по email)                    │
│         (Проверка качества выполненных работ)               │
└─────────────────────────────────────────────────────────────┘
```

#### 1.2. Принципы построения ролевой модели

| Принцип | Описание |
|---------|----------|
| **Минимальные привилегии** | Каждая роль имеет доступ только к необходимым для её функций данным и операциям |
| **Иерархичность** | ТМ управляет только своими инженерами и объектами, Администратор — всей системой |
| **Гибкость назначения** | Инженеры не привязаны жестко к объектам; любой инженер ТМ может работать на любом объекте территории ТМ |
| **Дублирование контроля** | И Инженер, и ТМ могут отправлять отчеты, что обеспечивает резервирование процесса |
| **Полный контроль истории** | Инженер и ТМ могут удалять визиты на любом этапе (до и после отправки) |

### 2. ДЕТАЛЬНОЕ ОПИСАНИЕ РОЛЕЙ

#### 2.1. АДМИНИСТРАТОР

**Описание:** Системный администратор, отвечающий за настройку системы, ведение справочников и управление доступом на уровне всей организации.

**Интерфейс доступа:** Web-интерфейс (десктопная версия) через браузер.

**Ключевые функции:**

##### Управление справочниками
- **CRUD адресов объектов** — добавление, редактирование, удаление адресов (город, улица, дом, строение, полный адрес)
- **Привязка email заказчиков** к адресам объектов
- **CRUD видов оборудования** — управление списком из 13 видов оборудования и привязанных контролируемыми параметрами
- **CRUD типовых рекомендаций** — ведение справочника ~80 рекомендаций с привязкой к видам оборудования
- **CRUD типов помещений** — управление списком типов помещений (12 типов)
- **Массовый импорт справочников через CSV** — загрузка адресов, видов оборудования, типов помещений, рекомендаций, пользователей, привязок

##### Управление пользователями и доступом
- **Создание учетных записей** инженеров, ТМ и других администраторов
- **Назначение объектов Территориальным менеджерам** — определение, какие объекты доступны каждому ТМ
- **Назначение инженеров в подчинение ТМ** — определение иерархии подчинения
- **Редактирование и блокировка** учетных записей
- **Возможность пересечения объектов** — один объект может быть закреплен за несколькими ТМ (на случай замены/дублирования)

##### Управление связью «Объект-Оборудование»
- **Просмотр оборудования объектов** — список всех единиц оборудования, привязанных к объекту
- **Добавление/редактирование оборудования** — с указанием марки, модели, серийного номера, описания местоположения
- **Массовый импорт привязки** через CSV с использованием аналитического кода объекта
- **Утверждение нового оборудования**, предложенного инженерами для включения в привязку

##### Глобальный контроль
- **Просмотр всех визитов** по всей системе (для аудита)
- **Формирование любых отчетов** (по визиту, сводных, по объекту)
- **Отправка отчетов** по email
- **Скачивание ZIP-архивов** любых визитов
- **Удаление любых визитов** в системе

**Ограничения:**
- Не создает визиты от своего имени (это функция Инженера)
- Не заполняет параметры оборудования (это полевая работа Инженера)

#### 2.2. ТЕРРИТОРИАЛЬНЫЙ МЕНЕДЖЕР (ТМ)

**Описание:** Диспетчер, контролирующий работу инженеров на закрепленной территории. Отвечает за качество отчетов, планирование работ и своевременную отправку результатов заказчикам.

**Интерфейс доступа:** Web-интерфейс (десктопная версия и/или планшет) через браузер.

**Ключевые функции:**

##### Управление доступом и объектами
- **Доступ к объектам:** Видит только объекты, закрепленные за ним Администратором
- **Гибкость доступа:** Любой инженер из группы ТМ может работать на любом объекте из территории ТМ
- **Возможность пересечения:** Один объект может быть закреплен за несколькими ТМ

##### Планирование работ
- **Создание визитов для инженеров** — ТМ может дистанционно создать визит и назначить его на конкретного подчиненного инженера
- **Указание планируемых даты и времени** (или оставление пустым)
- **Уведомление инженера** о назначенном визите

##### Контроль визитов
- **Просмотр визитов:** Видит все визиты всех подчиненных инженеров
- **Дашборд мониторинга:**
  - Количество запланированных/завершенных/в работе визитов
  - Статусы отправки отчетов
  - Уведомления о критических неисправностях
- **Переназначение:** Может переназначить незавершенный визит с одного инженера на другого из своей группы

##### Работа с отчетами
- **Просмотр:** Может просматривать любые визиты своих инженеров
- **Корректировка:** Может редактировать данные визита перед отправкой
- **Формирование отчетов:**
  - По каждому визиту отдельно
  - **Сводный отчет за период** (день/неделя/месяц)
  - **Отчет в разрезе объекта** (если работали несколько инженеров)
- **Отправка:** Может отправить отчет заказчику (дублирование не блокируется)
- **Скачивание:** Может скачать ZIP-архив с любого визита своих инженеров
- **Удаление:** Может удалять визиты своих инженеров на любом этапе (до и после отправки)

**Ограничения:**
- Не может редактировать справочники
- Не может просматривать визиты инженеров из других групп ТМ

---

#### 2.3. ИНЖЕНЕР

**Описание:** Полевой специалист, выполняющий техническое обслуживание оборудования на объектах.

**Интерфейс доступа:** Мобильное PWA-приложение на смартфоне или планшете.

**Ключевые функции:**

##### Полевая работа
- **Создание визитов** — выбор адреса объекта из справочника
- **Автоматическое получение списка оборудования** при выборе объекта
- **Выбор оборудования для обслуживания** из привязанного списка
- **Добавление непривязанного оборудования** с возможностью предложить включить в привязку
- **Заполнение параметров** — фиксация контролируемых параметров для каждого вида оборудования
- **Фотофиксация** — загрузка фото «ДО» и «ПОСЛЕ»
- **Работа офлайн** с последующей синхронизацией

##### Формирование и отправка отчетов
- **Формирование отчета** — генерация PDF и ZIP-архива
- **Отправка отчета** — на email заказчика
- **Резервное скачивание** ZIP-архива на устройство

##### Управление своими визитами
- **Просмотр только своих визитов**
- **Редактирование своих визитов** — до момента отправки
- **Удаление своих визитов** — на любом этапе (до и после отправки)
- **Получение назначенных визитов** от ТМ

**Особенности отправки:**
- Инженер может отправить отчет сразу после завершения визита
- После отправки Инженером ТМ не блокируется от повторной отправки
- В системе отображается статус отправки:
  - «Отправлен Инженером» (дата/время)
  - «Отправлен ТМ» (дата/время, если ТМ переотправлял)
  - «Не отправлен»

**Ограничения:**
- Может работать только на объектах, доступных его ТМ
- Не видит визиты других инженеров
- Не может редактировать справочники

---

#### 2.4. ЗАКАЗЧИК

**Описание:** Представитель организации-заказчика, получающий отчеты о выполненных работах.

**Интерфейс доступа:** Email-клиент (пассивная роль).

**Ключевые функции:**
- Получение email с ZIP-архивом (PDF + фотографии)
- Проверка качества выполненных работ
- Контроль по фотофиксации «ДО» и «ПОСЛЕ»

**Особенности:**
- Может получить несколько писем с одним отчетом (если и Инженер, и ТМ отправили)
- Email привязывается Администратором к адресу объекта

| **Откорректирован ТМ** | ТМ внес изменения в визит | ТМ |
| **Удален** | Визит удален (мягкое удаление) | Инженер / ТМ / Администратор |

### 5. ПРАВИЛА УДАЛЕНИЯ ВИЗИТОВ

#### 5.1. Кто может удалять

| Роль | Когда | Что удаляет |
|------|-------|-------------|
| **Инженер** | До отправки | Свой визит полностью |
| **Инженер** | После отправки | Свой визит полностью |
| **ТМ** | В любой момент | Любой визит своих инженеров |
| **Администратор** | В любой момент | Любой визит в системе |

#### 5.2. Логика удаления

1. **Необратимость Email:** При удалении визита система не пытается отозвать письмо из почтового ящика Заказчика.

2. **Мягкое удаление (Soft Delete):** В базе данных визит помечается флагом `is_deleted = true`, но данные сохраняются для аудита.

3. **Отображение:**
   - Удаленный визит исчезает из активного списка и дашборда
   - В разделе «Архив / Удаленные визиты» (доступен ТМ и Администратору) сохраняется с пометкой *«Удален [Кем] [Когда]»*

4. **Влияние на сводные отчеты:** Удаленные визиты не включаются в статистику и PDF-документ.

## ЧАСТЬ II. МОДУЛЬ МАССОВОГО ИМПОРТА СПРАВОЧНИКОВ (CSV)

### 6. ОБЩИЕ ТРЕБОВАНИЯ К ИМПОРТУ

#### 6.1. Процесс импорта

```
┌─────────────────────────────────────────────────────────────┐
│  ИМПОРТ СПРАВОЧНИКА: [Выбрать справочник ▼]                 │
├─────────────────────────────────────────────────────────────
│                                                             │
│   Файл: [Выбрать файл...]  addresses_2026.csv              │
│  📊 Размер: 2.4 МБ · 15 432 строк                          │
│                                                             │
│  ⚙️ НАСТРОЙКИ ИМПОРТА:                                      │
│  ☑ Предварительная проверка (рекомендуется)                │
│  ☐ Обновить существующие записи                            │
│                                                             │
│  [🔍 Проверить файл]  [ Загрузить]  [❌ Отмена]          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 6.2. Режимы обработки данных

**Режим 1: Предварительная проверка (по умолчанию)**
- Система анализирует CSV без изменения данных
- Показывает отчёт:
  - ✅ Корректных строк
  - ️ Дубликатов (будут пропущены)
  - ❌ Ошибочных строк (с указанием номеров и причин)
- Пользователь подтверждает загрузку или корректирует файл

**Режим 2: Прямая загрузка**
- Загрузка без предварительной проверки
- Корректные строки загружаются, ошибочные пропускаются
- В конце показывается итоговый отчёт

#### 6.3. Обработка дубликатов

**Правило:** Дубликаты **пропускаются** (не обновляются и не создаются повторно).

**Логика определения дубликата:**
- Для адресов: совпадение `full_address`
- Для оборудования: совпадение `code`
- Для пользователей: совпадение `email`
- Для привязок: совпадение комбинации ключевых полей

#### 6.4. Обработка ошибок

| Тип ошибки | Пример | Действие системы |
|------------|--------|------------------|
| **Незаполненное обязательное поле** | Пустой `full_address` | Пропуск строки, запись в отчёт |
| **Некорректный формат email** | `user@domain` (без точки) | Пропуск строки, запись в отчёт |
| **Некорректное числовое значение** | `photos_required = 5` | Пропуск строки, запись в отчёт |
| **Нарушение уникальности** | Дубликат `code` | Пропуск строки, запись в отчёт |
| **Ссылка на несуществующую запись** | `equipment_type = "Несуществующий тип"` | Пропуск строки, запись в отчёт |
| **Кодировка файла** | Нечитаемые символы | Ошибка загрузки всего файла |

#### 6.5. Лимиты и производительность

| Параметр | Значение |
|----------|----------|
| **Максимальное количество строк** | 20 000 |
| **Максимальный размер файла** | 10 МБ |
| **Поддерживаемые кодировки** | UTF-8, Windows-1251 |
| **Разделитель** | Запятая (`,`), точка с запятой (`;`) — автоопределение |
| **Индикатор прогресса** | Обязателен (процент + количество обработанных строк) |
| **Ограничение частоты** | Нет |

## ЧАСТЬ IV. USER STORIES И СЦЕНАРИИ

### 13. НОВЫЕ USER STORIES

#### 13.1. Модуль массового импорта

**US-IMP-1 — Импорт справочника адресов**  
*Как* администратор, *я хочу* загрузить CSV-файл со списком адресов объектов, *чтобы* быстро наполнить справочник при первичном внедрении системы.

**US-IMP-2 — Предварительная проверка CSV**  
*Как* администратор, *я хочу* перед загрузкой файла увидеть отчёт о корректных, дублирующихся и ошибочных строках, *чтобы* исправить ошибки до импорта.

**US-IMP-3 — Обработка дубликатов при импорте**  
*Как* система, *я хочу* автоматически пропускать дублирующиеся записи при импорте, *чтобы* не создавать дубли в справочнике.

**US-IMP-4 — Импорт пользователей с привязкой к ТМ**  
*Как* администратор, *я хочу* загрузить CSV со списком инженеров и ТМ с указанием привязки инженеров к ТМ, *чтобы* сразу настроить иерархию подчинения.

**US-IMP-5 — Генерация временных паролей**  
*Как* система, *я хочу* генерировать временный общий пароль для пользователей, созданных через импорт, *чтобы* администратор не задавал пароли вручную.

**US-IMP-6 — Индикатор прогресса импорта**  
*Как* администратор, *я хочу* видеть прогресс загрузки файла в реальном времени, *чтобы* понимать, сколько времени осталось до завершения.

**US-IMP-7 — История импортов**  
*Как* администратор, *я хочу* видеть историю всех импортов с датами и результатами, *чтобы* отслеживать изменения в справочниках.

**US-IMP-8 — Импорт привязки «Объект-Оборудование»**  
*Как* администратор, *я хочу* загрузить CSV со списком оборудования для каждого объекта, *чтобы* настроить связь между объектами и оборудованием.

---

#### 13.2. Модуль связи «Объект-Оборудование»

**US-OBJ-1 — Просмотр оборудования объекта**  
*Как* администратор, *я хочу* видеть список всего оборудования, привязанного к объекту, *чтобы* контролировать полноту данных.

**US-OBJ-2 — Добавление оборудования к объекту**  
*Как* администратор, *я хочу* добавлять новые единицы оборудования к объекту с указанием марки, модели, серийного номера и описания местоположения, *чтобы* вести полный учёт оборудования.

**US-OBJ-3 — Автоматическое заполнение чек-листа**  
*Как* инженер, *я хочу*, чтобы при выборе адреса объекта чек-лист автоматически заполнялся списком привязанного оборудования, *чтобы* не добавлять каждую единицу вручную.

**US-OBJ-4 — Выбор оборудования для визита**  
*Как* инженер, *я хочу* отмечать галочками только то оборудование, которое буду обслуживать в текущем визите, *чтобы* не выполнять лишнюю работу.

**US-OBJ-5 — Самостоятельный выбор помещения**  
*Как* инженер, *я хочу* самостоятельно выбирать тип помещения для каждой единицы оборудования при создании визита, *чтобы* уточнить фактическое местоположение.

**US-OBJ-6 — Добавление непривязанного оборудования**  
*Как* инженер, *я хочу* иметь возможность добавить оборудование, которого нет в привязке объекта, *чтобы* обслужить новое или забытое оборудование.

**US-OBJ-7 — Предложение добавить в привязку**  
*Как* инженер, *я хочу* отметить добавленное оборудование для последующего включения в привязку объекта, *чтобы* администратор мог утвердить его.

**US-OBJ-8 — Утверждение нового оборудования**  
*Как* администратор, *я хочу* видеть список оборудования, предложенного инженерами для добавления в привязку, *чтобы* утверждать или отклонять запросы.

### 15. ТЕХНИЧЕСКИЕ ТРЕБОВАНИЯ К ИМПОРТУ

#### 15.1. Парсинг CSV

**Библиотеки:**
- **Backend (Python):** `pandas`, `csv` (стандартная библиотека)
- **Backend (Node.js):** `csv-parser`, `papaparse`
- **Frontend (валидация):** `papaparse` (для предварительной проверки на клиенте)

**Требования к парсеру:**
- Автоопределение разделителя (`,` или `;`)
- Автоопределение кодировки (UTF-8, Windows-1251)
- Поддержка кавычек для полей с разделителями внутри
- Обработка пустых строк (пропуск)
- Обработка комментариев (строки, начинающиеся с `#`)

#### 15.2. Валидация данных

**Серверная валидация (обязательная):**
- Проверка обязательных полей
- Проверка форматов (email, числа, даты)
- Проверка уникальности
- Проверка ссылок на существующие записи (FK)
- Проверка бизнес-логики (например, `photos_required` только 1 или 2)

**Клиентская валидация (опциональная, для UX):**
- Предварительная проверка файла перед загрузкой
- Показ ошибок в реальном времени
- Подсветка проблемных строк

#### 15.3. Производительность

**Требования:**
- Обработка 20 000 строк: не более 60 секунд
- Память: не более 512 МБ на процесс импорта
- Использование транзакций БД (откат при критических ошибках)
- Пакетная вставка данных (batch insert) для ускорения

**Оптимизация:**
- Индексы на полях уникальности
- Отключение триггеров на время массового импорта (если возможно)
- Использование `COPY` (PostgreSQL) или аналогов для быстрой вставки

#### 15.4. Безопасность

**Требования:**
- Проверка типа файла (только `.csv`)
- Проверка размера файла (макс. 10 МБ)
- Санитизация данных (защита от CSV-инъекций)
- Логирование всех операций импорта (кто, когда, что загружал)
- Ограничение доступа: только роль Администратор

- `RU` — фиксированный префикс (Россия)

- `sequential_number` — порядковый номер объекта в регионе (3 цифры, с ведущими нулями)
