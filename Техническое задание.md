# ТЕХНИЧЕСКОЕ ЗАДАНИЕ
## на разработку web-приложения «Цифровой чек-лист инженера по техническому обслуживанию оборудования»

---

## 1. ОБЩИЕ СВЕДЕНИЯ

### 1.1. Наименование системы
Web-приложение «Цифровой чек-лист инженера по техническому обслуживанию оборудования» (далее — Приложение).

### 1.2. Назначение системы
Автоматизация деятельности инженера по техническому обслуживанию оборудования: создание чек-листа работ, фиксация контролируемых параметров, фотофиксация состояния оборудования «до» и «после» работ, формирование отчета в формате PDF с маркированными фотографиями и отправка заказчику по email.

### 1.3. Цели проекта
- Оцифровать процесс обслуживания оборудования
- Подтвердить факт выполнения работ фотофиксацией
- Сократить трудозатраты инженера на 30–40%
- Исключить влияние человеческого фактора на полноту отчетов
- Обеспечить работу в условиях отсутствия интернета (подвалы, серверные)
- Не мешать посетителям объекта во время работ

### 1.4. Нормативные ссылки
- ФЗ-152 «О персональных данных»
- ГОСТ Р ИСО/МЭК 12207-2010 «Программная инженерия. Процессы жизненного цикла программных средств»
- СП 60.13330.2020 «Отопление, вентиляция и кондиционирование воздуха»
- ПУЭ (Правила устройства электроустановок)

## 4. ФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ

### 4.1. Модуль «Инженер»

#### 4.1.1. Создание визита

**US-1.1 — Выбор адреса из справочника**  
*Как* инженер, *я хочу* выбирать адрес объекта из справочника с функцией поиска/автодополнения, *чтобы* избежать ошибок ввода и ускорить начало работы.

**US-1.2 — Автозаполнение даты и времени**  
*Как* инженер, *я хочу*, чтобы поля «Дата начала» и «Время начала» заполнялись автоматически текущими значениями, *чтобы* не вводить их вручную.

**US-1.3 — Автоопределение сезона**  
*Как* инженер, *я хочу*, чтобы поле «Сезон» автоматически определялось по дате визита:
- 01.04–31.10 → «Лето»
- 01.11–31.03 → «Зима»

*чтобы* исключить расхождения в отчетах.  
Поле доступно для ручного редактирования. При изменении даты — пересчет с подтверждением.

**US-1.4 — Добавление оборудования через модальное окно**
*Как* инженер, *я хочу* нажимать «+ Добавить оборудование» и выбирать нужные единицы оборудования из модального окна с тремя вкладками («Уровень помещения», «Уровень объекта» и «Добавить новое»), *чтобы* формировать состав визита пошагово, без дубликатов и визуального шума.
При первом нажатии кнопки для несохранённого визита система автоматически сохраняет визит как черновик, затем открывает модальное окно.

Вкладки модального окна:
- **«Уровень помещения»** — для оборудования с привязкой к помещению (климатическое оборудование: СС, МСС, VRV, тепловые завесы, приточные и приточно-вытяжные установки и т.д.). Инженер выбирает помещение из списка, затем видит всё оборудование в этом помещении (все чекбоксы отмечены по умолчанию) и нажимает «Добавить».
- **«Уровень объекта»** — для оборудования без привязки к помещению (сети, тепловые сети, приборы учёта, РЩ без помещения). Список с чекбоксами, все отмечены по умолчанию.
- **«Добавить новое»** — ручной ввод оборудования, которого нет в привязке объекта.

Уровень привязки определяется по наличию `roomTypeCode` в записи `object_equipment`, а не по коду типа оборудования.

**US-1.5 — Визуальный статус задач**  
*Как* инженер, *я хочу* видеть в чек-листе для каждой задачи:
- Статус: «Не начато» / «В работе» / «Выполнено»
- Прогресс фото: ⬜ / 📷 1/2 / 📷 2/2

*чтобы* понимать, что уже сделано.

**US-1.6 — Возврат к незавершенной задаче**
*Как* инженер, *я хочу* иметь возможность кликнуть на любую незавершенную задачу и продолжить её заполнение, *чтобы* не начинать работу заново.

**US-1.7 — Работа офлайн**
*Как* инженер, *я хочу* создавать визиты и загружать фото без подключения к интернету, *чтобы* работать в подвалах, серверных и других зонах без связи.
Данные сохраняются локально и автоматически синхронизируются при восстановлении соединения.

**US-1.8 — Выбор оборудования из привязки объекта через модальное окно (группировка по уровням)**
*Как* инженер, *я хочу*, чтобы при нажатии «+ Добавить оборудование» оборудование из привязки объекта (`object_equipment`) отображалось сгруппированным по уровню привязки — с помещениями и без, *чтобы* выбирать нужные позиции осознанно, без дубликатов.

Система загружает список оборудования из `object_equipment` при открытии модального окна. Уже добавленные в визит единицы скрыты из списка. При повторном открытии модалки отображаются только оставшиеся. Если всё оборудование уже добавлено — соответствующая вкладка показывает сообщение «Всё оборудование уже добавлено в визит».

**Группировка по уровням привязки:**
- **Уровень помещения** — оборудование, у которого указан `roomTypeCode` в `object_equipment`. Инженер сначала выбирает помещение из выпадающего списка, затем видит все единицы оборудования в этом помещении. Все чекбоксы отмечены по умолчанию. После нажатия «Добавить» выбранные единицы добавляются в визит.
- **Уровень объекта** — оборудование без привязки к помещению (`roomTypeCode` отсутствует). Отображается плоским списком с чекбоксами, все отмечены по умолчанию.

**Уточнение помещения:**
Для единиц оборудования, у которых не указан тип помещения, отображается метка «⚠️ Помещение не указано». При выборе такого оборудования появляется inline-список для выбора типа помещения из справочника. После добавления визита выбранное помещение сохраняется в `object_equipment` (поля `room_confirmed_at`, `room_confirmed_by`) и используется во всех последующих визитах.

**Новый API-эндпоинт:**
`GET /api/refs/object-equipment/rooms?address_id=...&exclude_visit_id=...` — возвращает список помещений с количеством оборудования для текущего объекта (исключая уже добавленные в указанный визит).

**US-1.9 — Предложение добавить оборудование в привязку объекта**
*Как* инженер, *я хочу* отметить чекбоксом «Предложить добавить в привязку объекта» при добавлении оборудования, *чтобы* предложить администратору закрепить обнаруженное оборудование за данным объектом для будущих визитов.
Предложение сохраняется со статусом `pending` и становится доступным администратору для утверждения или отклонения.

**US-GROUP-01 — Группировка оборудования по уровню помещения**
*Как* инженер, *я хочу* выбирать оборудование с привязкой к помещению, предварительно выбирая нужное помещение из списка, *чтобы* быстро находить оборудование в конкретных помещениях объекта.
На вкладке «📍 Уровень помещения» инженер видит выпадающий список помещений (полученный через `GET /api/refs/object-equipment/rooms`). После выбора помещения отображается чекбокс-лист всех единиц оборудования в этом помещении. Все чекбоксы отмечены по умолчанию. Уровень привязки определяется по наличию `roomTypeCode` в записи `object_equipment`.

**US-GROUP-02 — Группировка оборудования по уровню объекта**
*Как* инженер, *я хочу* видеть оборудование без привязки к помещению отдельным списком, *чтобы* добавлять его в визит без необходимости указывать помещение.
На вкладке «🏠 Уровень объекта» отображается плоский чекбокс-лист единиц оборудования из `object_equipment`, у которых `roomTypeCode` отсутствует (сети, тепловые сети, приборы учёта, РЩ без помещения). Все чекбоксы отмечены по умолчанию.

**US-METER-01 — Автозаполнение полей счётчиков из справочника объекта**
*Как* инженер, *я хочу*, чтобы при открытии задачи-счётчика (Прибор учёта э/э, ХВС, ГВС), созданной из связанного оборудования (`object_equipment`), поля «Модель счётчика» и «Номер счётчика» автоматически заполнялись из данных привязки, *чтобы* не вводить повторяющиеся данные вручную.
Логика автозаполнения:
- «Модель счётчика» = `brand` + " " + `model` (конкатенация; если `brand` пуст — только `model`)
- «Номер счётчика» = `serial_number`
Поля остаются редактируемыми после автозаполнения. Рядом с автозаполненными полями отображается иконка 📋 с тултипом «Автозаполнено из справочника объекта».

#### 4.1.6. Группировка климатического оборудования в единый чек-лист

**US-GROUP-NEW-01 — Групповой чек-лист для климата в помещении**
*Как* инженер ВиК, *я хочу*, чтобы при добавлении климатического оборудования в помещении формировался один групповой чек-лист, *чтобы* не заполнять одинаковые параметры (температура помещения, заключение) для каждой единицы отдельно.
Система создаёт одну задачу типа `group_climate` на все внутренние блоки климатического оборудования (СС, МСС, VRV) в помещении. Общие параметры (температура, заключение, рекомендации) заполняются один раз. Для каждой единицы оборудования инженер указывает индивидуальный статус (ОК/Не ОК) и загружает фото ДО/ПОСЛЕ.

**US-GROUP-NEW-02 — Групповой чек-лист для наружных блоков**
*Как* инженер, *я хочу*, чтобы наружные блоки кондиционеров (splitnar, mssnar, vrv_nar) группировались в отдельную задачу по уровню объекта (здания), *чтобы* не привязывать их к конкретному помещению.
Наружные блоки определяются по полю `is_outdoor_unit = true` в `object_equipment`. Все наружные блоки объекта формируют одну групповую задачу.

**US-GROUP-NEW-03 — Индивидуальные статусы и фото для каждой единицы**
*Как* инженер, *я хочу* для каждой единицы в групповом чек-листе указывать индивидуальный статус (ОК/Не ОК) и загружать 2 фото (до/после), *чтобы* фиксировать состояние каждого кондиционера отдельно.

**US-GROUP-NEW-04 — Добавление и удаление единиц в групповой задаче**
*Как* инженер, *я хочу* иметь возможность добавлять и удалять единицы оборудования в групповом чек-листе, *чтобы* гибко формировать состав работ.

**US-GROUP-NEW-05 — Валидация и сохранение групповой задачи**
*Как* система, *я хочу* разрешать сохранение групповой задачи с частичным заполнением, *чтобы* инженер мог сохранить прогресс и вернуться к незавершённым единицам позже.
Логика сохранения:
- Инженер может нажать «Сохранить» в любой момент — общие параметры и заполненные данные единиц сохраняются
- Статус задачи определяется автоматически:
  - `in_progress` («В работе») — если хотя бы одна единица не имеет статуса или фото
  - `completed` («Выполнено») — если все единицы имеют статус и оба фото (ДО и ПОСЛЕ)
- При сохранении неполной задачи отображается сообщение: «Сохранено. Не завершено единиц: N»
- Если хотя бы одна единица имеет статус «Не ОК», поле «Дополнительные рекомендации» становится обязательным

**US-GROUP-NEW-06 — Единообразие структуры при одной единице**
*Как* система, *я хочу* даже при наличии только одной единицы климатического оборудования в помещении формировать групповую задачу, *чтобы* обеспечить единообразие структуры данных.

**Принцип группировки задач:**

| Тип задачи | Оборудование | Уровень | Формирование |
|------------|-------------|---------|--------------|
| **Групповая (климат)** | Внутренние блоки СС, МСС, VRV | Помещение | Один чек-лист на все единицы в помещении |
| **Групповая (наружные)** | Наружные блоки СС, МСС, VRV | Объект | Один чек-лист на все наружные блоки объекта |
| **Индивидуальная** | Счётчики, РЩ/ГРЩ, сети, вентиляция | По единице | Отдельный чек-лист на каждую единицу |

#### 4.1.2. Заполнение параметров

**US-2.1 — Быстрый ввод рекомендаций**  
*Как* инженер, *я хочу* отмечать галочками типовые рекомендации из справочника (с фильтрацией по виду оборудования), *чтобы* не печатать текст на телефоне в полевых условиях.

**US-2.2 — Условная обязательность**  
*Как* система, *я хочу* делать поле «Дополнительные рекомендации» обязательным только при выборе заключения «Исправно, есть замечания» или «Неисправно», *чтобы* инженер описывал проблему, но не тратил время, если всё в норме.

**US-2.3 — Безопасное редактирование**  
*Как* инженер, *я хочу*, чтобы кнопка «Сбросить» очищала **только текущую задачу**, а не весь визит, *чтобы* случайное нажатие не уничтожило уже проделанную работу.

#### 4.1.3. Фотофиксация

**US-3.1 — Фото ДО и ПОСЛЕ за один проход**  
*Как* инженер, *я хочу* на Странице 5 сразу загружать оба фото (до и после работ), *чтобы* зафиксировать результат без возврата в чек-лист.

**US-3.2 — Разное количество фото**
*Как* система, *я хочу* требовать:

### 4.2. Модуль «Администратор»

**US-ADM-1 — CRUD справочника адресов**
*Как* администратор, *я хочу* добавлять, редактировать и удалять адреса (город, улица, дом, строение, полный адрес), *чтобы* инженер выбирал из списка.
Удаление адресов выполняется как **soft delete** (пометка `is_deleted`) — адрес скрывается из списков и поиска, но сохраняется для исторических визитов и связанных записей.

**US-ADM-2 — CRUD справочника рекомендаций**
*Как* администратор, *я хочу* вести справочник типовых рекомендаций с привязкой к виду оборудования (17 активных категорий + 1 деактивированная), *чтобы* инженер выбирал их чек-боксами.

**US-ADM-3 — CRUD видов оборудования**
*Как* администратор, *я хочу* управлять списком видов оборудования (17 активных + 1 деактивированный) и привязанными контролируемыми параметрами, *чтобы* гибко настраивать приложение.
Каждый вид оборудования имеет поле `specialization_req` (TEXT, nullable) — требуемая специализация (`vik` или `iszh`), определяющее, для какой специализации инженера доступно данное оборудование.

**US-ADM-4 — CRUD типов помещений**  
*Как* администратор, *я хочу* управлять списком типов помещений (Электрощитовая, Серверная, Клиентский зал и т.д.).

**US-ADM-5 — Привязка email к адресу**  
*Как* администратор, *я хочу* указывать email заказчика при создании/редактировании объекта, *чтобы* инженер мог сразу отправить отчет.

**US-ADM-6 — Управление пользователями**
*Как* администратор, *я хочу* создавать, редактировать и блокировать учетные записи инженеров с привязкой ФИО.

**US-ADM-7 — Массовый импорт справочников**
*Как* администратор, *я хочу* загружать справочники (адреса, оборудование, рекомендации, пользователи) из файлов форматов CSV и XLSX, *чтобы* быстро наполнять систему начальными данными без ручного ввода каждой записи.
Поддерживаемые возможности:
- Поддержка форматов: `.csv` (разделители `,` и `;`, кодировки UTF-8 / Windows-1251) и `.xlsx` (Microsoft Excel 2007+)
- Для Excel-файлов: данные считываются с первого листа, первая строка — заголовки столбцов, пустые строки игнорируются
- Предварительная валидация (pre-validation) — проверка формата и обязательных полей до импорта с отображением списка ошибок
- Автоопределение кодировки файла (UTF-8 / Windows-1251) для CSV
- Ограничение на количество строк: до 20 000 за один импорт
- Ограничение на размер файла: до 10 МБ
- Для импорта пользователей — автоматическая генерация временных паролей и отправка их на email пользователю с уведомлением об учётной записи
- Rate limiting на эндпоинтах аутентификации: не более 20 запросов за 15 минут для защиты от брутфорса

**US-ADM-8 — Управление привязкой «Объект-Оборудование»**
*Как* администратор, *я хочу* управлять привязкой оборудования к объектам (адресам): добавлять, редактировать, деактивировать записи, *чтобы* инженер при создании визита выбирал оборудование для известных объектов.
Интерфейс позволяет:
- Выбрать адрес из справочника и увидеть список привязанного оборудования
- Добавить новую единицу оборудования с указанием марки, модели, серийного номера, типа помещения (опционально) и описания расположения
- Деактивировать оборудование (без физического удаления — для сохранения истории визитов)
- Записи без указанного помещения отображаются с пометкой «⚠️ Не указано»
- При импорте оборудования поле «Тип помещения» является опциональным — оборудование может быть привязано к объекту без указания конкретного помещения

**US-ADM-9 — Утверждение предложенного оборудования**
*Как* администратор, *я хочу* просматривать и утверждать/отклонять предложения инженеров о добавлении оборудования в привязку объекта, *чтобы* контролировать состав привязанного оборудования и учитывать полевые наблюдения инженеров.
Интерфейс показывает список предложений со статусом `pending`: инженер, адрес, предложенное оборудование. Администратор может утвердить (оборудование добавляется в `object_equipment` со статусом `is_active = true`) или отклонить (статус предложения → `rejected`) с комментарием.

| Чекбокс | Вид оборудования | Помещение | Марка / Модель | Вложения | Статус |
|:-------:|------------------|-----------|----------------|:--------:|--------|

**Кнопки:**
- `+ Добавить оборудование` — открывает модальное окно «Добавление оборудования» с тремя вкладками:
  - **Вкладка «📍 Уровень помещения»** — для оборудования с привязкой к помещению (определено по наличию `roomTypeCode` в `object_equipment`). Инженер выбирает помещение из списка (полученного через `GET /api/refs/object-equipment/rooms`), затем видит всё оборудование в этом помещении в виде чекбокс-листа. Все чекбоксы отмечены по умолчанию. Уже добавленные в визит единицы скрыты. Инженер нажимает «Добавить» — для каждой выбранной записи создаётся задача с автоматическим заполнением полей (тип оборудования, помещение, марка, модель, серийный номер, расположение).
  - **Вкладка «🏠 Уровень объекта»** — для оборудования без привязки к помещению (`roomTypeCode` отсутствует). Отображается чекбокс-лист единиц оборудования из `object_equipment`, отфильтрованных по специализации инженера. Все чекбоксы отмечены по умолчанию. Уже добавленные в визит единицы скрыты.
  - **Вкладка «➕ Добавить новое»** — форма для добавления оборудования, которого нет в привязке объекта:
    - Вид оборудования (выпадающий список из справочника)
    - Тип помещения (выпадающий список из справочника)
    - Комментарий (текстовое поле, необязательно)
    - Марка (текстовое поле, необязательно)
    - Модель (текстовое поле, необязательно)
    - Серийный номер (текстовое поле, необязательно)
    - Чекбокс «Предложить добавить в привязку объекта» (если отмечен — при сохранении создаётся предложение со статусом `pending` для утверждения администратором)
- `Завершить визит` — формирует PDF + ZIP (активна при наличии хотя бы 1 выполненной задачи)

**Логика:**
- При открытии нового визита таблица «Проведённые работы» **полностью пустая**. Никакое оборудование автоматически не подтягивается.
- Кнопка `+ Добавить оборудование` доступна сразу. Для нового (несохранённого) визита — при нажатии визит автоматически сохраняется как черновик, затем открывается модальное окно.
- При повторном нажатии «+ Добавить оборудование» во вкладках «Уровень помещения» и «Уровень объекта» отображаются только те единицы, которые инженер ещё не добавил в текущий визит.
- Если всё оборудование определённого уровня уже добавлено, соответствующая вкладка показывает сообщение «Всё оборудование уже добавлено в визит».
- Клик на строку таблицы открывает соответствующую страницу параметров (4.x)
- Кнопка «Завершить визит» неактивна, пока хотя бы одна задача не выполнена

### 6.6. Привязка «Объект-Оборудование» (object_equipment)

Модуль хранит информацию о том, какое оборудование физически установлено на каждом объекте (адресе). Используется для автозаполнения чек-листа инженера при создании визита.

**Структура таблицы:**

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| address_id | UUID | FK на справочник адресов |
| equipment_type_code | VARCHAR(50) | Код вида оборудования |
| room_type_code | VARCHAR(50) | Код типа помещения (**опционально**, может быть NULL) |
| brand | VARCHAR(200) | Марка оборудования (опционально) |
| model | VARCHAR(200) | Модель оборудования (опционально) |
| serial_number | VARCHAR(200) | Серийный номер (опционально) |
| location_description | TEXT | Описание расположения на объекте (опционально) |
| is_active | BOOLEAN | Активна ли привязка (true по умолчанию) |
| room_confirmed_at | TIMESTAMP | Дата первичного уточнения помещения инженером (опционально) |
| room_confirmed_by | UUID | FK на пользователя, уточнившего помещение (опционально) |
| created_at | TIMESTAMP | Дата создания |
| updated_at | TIMESTAMP | Дата обновления |

**Уникальные ограничения:**
- `(address_id, equipment_type_id, serial_number)` — одна и та же единица оборудования не может быть привязана к объекту дважды
- Частичный индекс: ограничение действует только для записей с `is_active = true` и непустым `serial_number`

**Поведение при добавлении оборудования в визит:**
- При нажатии кнопки «+ Добавить оборудование» открывается модальное окно с тремя вкладками: «📍 Уровень помещения», «🏠 Уровень объекта», «➕ Добавить новое»
- Оборудование распределяется по вкладкам на основании наличия `room_type_code` в записи `object_equipment`:
  - Записи с указанным `room_type_code` → вкладка «Уровень помещения» (группируются по помещению)
  - Записи без `room_type_code` → вкладка «Уровень объекта» (плоский список)
- На обеих вкладках отображаются только активные записи `object_equipment` для данного `address_id`, отфильтрованные по специализации инженера
- Записи, уже добавленные в текущий визит (через поле `object_equipment_id` в таблице `tasks`), **исключаются** из списка — это предотвращает дубликаты
- Список помещений для вкладки «Уровень помещения» загружается через `GET /api/refs/object-equipment/rooms?address_id=...&exclude_visit_id=...`
- Для записей без указанного помещения (`room_type_code = NULL`) отображается метка «⚠️ Помещение не указано»
- При выборе таких записей появляется inline-список для выбора типа помещения из справочника
- Кнопка «Добавить» неактивна, пока для всех отмеченных единиц без помещения не выбран тип
- При добавлении: для оборудования без помещения вызывается `PATCH /api/refs/object-equipment/:id/room` — помещение сохраняется в `object_equipment` с заполнением `room_confirmed_at` и `room_confirmed_by`
- Инженер отмечает галочками нужные позиции и нажимает «Добавить» — для каждой создаётся задача с предзаполненными полями
- Если все записи уже добавлены, соответствующая вкладка показывает сообщение «Всё оборудование уже добавлено в визит»
- На вкладке «➕ Добавить новое» инженер может добавить оборудование вручную (не из привязки)
- Новые строки, добавленные вручную, не сохраняются в `object_equipment` автоматически (для этого инженер отмечает чекбокс «Предложить добавить в привязку объекта»)

### 6.7. Предложения оборудования (equipment_proposals)

Модуль хранит предложения инженеров о добавлении нового оборудования в привязку «Объект-Оборудование». Предложения создаются, когда инженер обнаруживает на объекте оборудование, которое ещё не привязано.

**Структура таблицы:**

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| address_id | UUID | FK на справочник адресов |
| equipment_type_id | UUID | FK на справочник видов оборудования |
| room_type_id | UUID | FK на справочник типов помещений (опционально) |
| brand | VARCHAR(200) | Марка оборудования |
| model | VARCHAR(200) | Модель оборудования |
| serial_number | VARCHAR(200) | Серийный номер |
| location_description | TEXT | Описание расположения |
| proposed_by_id | UUID | FK на пользователя (инженер), предложившего оборудование |
| status | VARCHAR(20) | Статус предложения: `pending`, `approved`, `rejected` |
| reviewed_by_id | UUID | FK на пользователя (администратор), рассмотревшего предложение (опционально) |
| review_comment | TEXT | Комментарий администратора при утверждении/отклонении |
| reviewed_at | TIMESTAMP | Дата рассмотрения |
| created_at | TIMESTAMP | Дата создания предложения |

**Жизненный цикл:**
1. Инженер добавляет оборудование в визит и отмечает чекбокс «Предложить добавить в привязку объекта» → создаётся запись со статусом `pending`
2. Администратор видит предложение в списке `pending`-предложений
3. Администратор утверждает → статус меняется на `approved`, запись автоматически копируется в `object_equipment` с `is_active = true`
4. Администратор отклоняет → статус меняется на `rejected`, указывается причина в `review_comment`

### 6.8. Маппинг специализаций

Каждый инженер имеет одну или обе специализации, определяющие набор обслуживаемого оборудования.

| Специализация | Поле в БД | Коды оборудования | Описание |
|:-------------:|-----------|-------------------|----------|
| **ВиК** | `specialization_vik` | ~~vent~~ (деактивирован), teplozavesa, pritochnaya, pritochno-vytyzhnaya, vytyzhnaya, vrv_vn, vrv_nar, mssvn, mssnar, splitvn, splitnar | Вентиляция и кондиционирование (11 активных типов + 1 деактивированный) |

**Правила:**
- Значения по умолчанию при создании инженера: `specialization_vik = false`, `specialization_iszh = true`

- ТМ/Администратор могут изменить специализацию подчинённого инженера
- Если обе специализации сброшены (false/false) — показывается блокирующий экран (SpecializationGate)
- Задачи в визите фильтруются по специализации инженера
- Оборудование объекта фильтруется по специализации при автозаполнении
- Специализация инженера отображается в PDF-отчёте

### 8.2.1. Офлайн-режим

- Фотографии хранятся как Blob в IndexedDB до момента синхронизации
- Очередь мутаций с повторными попытками (до 5 раз)
- Визуальный индикатор статуса: «Офлайн» / «Синхронизация...» / «Онлайн»
- Service Worker кэширует API-ответы (NetworkFirst, timeout 3с, TTL 7 дней)

**Технологии:**
- Dexie.js (IndexedDB wrapper)

### 9.2. Основные таблицы

**Users** — пользователи системы. Поле `role` имеет значения: `engineer`, `tm`, `admin`.
**Addresses** — справочник адресов
**RoomTypes** — типы помещений
**EquipmentTypes** — виды оборудования
**Recommendations** — типовые рекомендации
**Visits** — визиты инженера. Дополнительные поля: `assigned_by_id` (UUID — кем назначен), `assigned_at` (TIMESTAMP — дата назначения), `sent_by_engineer_at` (TIMESTAMP — дата отправки инженером), `sent_by_tm_at` (TIMESTAMP — дата отправки ТМ), `tm_corrected` (BOOLEAN — откорректирован ТМ), `is_deleted` (BOOLEAN — удалён ли визит), `deleted_by_id` (UUID — кем удалён), `deleted_at` (TIMESTAMP — дата удаления).

Поле `status` (VisitStatus) принимает следующие значения:
- `planned` — Запланировано (ТМ назначил визит инженеру)
- `not_started` — Не начато
- `in_progress` — В работе
- `completed` — Выполнено
- `sent_by_engineer` — Отправлен инженером
- `sent_by_tm` — Отправлен ТМ
- `corrected_by_tm` — Откорректирован ТМ

**Tasks** — задачи в чек-листе. Поля: `task_type` (VARCHAR, default 'individual' — тип задачи: 'individual' или 'group_climate'), `object_equipment_id` (UUID, nullable — ссылка на `object_equipment`, если задача создана из привязки объекта), `room_type_code` (VARCHAR, nullable — код помещения для групповых задач), `comment` (текстовый комментарий, бывшее `location`), `brand` (марка оборудования), `model` (модель оборудования), `serial_number` (серийный номер). Для групповых задач (`task_type = 'group_climate'`) общие параметры хранятся в `parameters`, а единицы оборудования — в связанной таблице `task_equipment_items`.
**Photos** — привязка фото к задачам. Для индивидуальных задач — через `task_id`. Для групповых задач — через `task_equipment_item_id` (каждое фото привязано к конкретной единице в групповой задаче).

**Таблица task_equipment_items** — единицы оборудования в групповой задаче:

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| task_id | UUID | FK на tasks (групповая задача) |
| object_equipment_id | UUID | FK на object_equipment (конкретная единица) |
| status | VARCHAR | Статус единицы: 'ok' / 'not_ok' (nullable) |
| sort_order | INT | Порядок отображения |
| created_at | TIMESTAMP | Дата создания |

Уникальный: `(task_id, object_equipment_id)`.

**Таблица tm_objects** — привязка объектов к ТМ:

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| tm_id | UUID | Ссылка на ТМ |
| address_id | UUID | Ссылка на адрес объекта |
| created_at | TIMESTAMP | Дата создания |

**Таблица tm_engineers** — привязка инженеров к ТМ:

| Поле | Тип | Описание |
|------|-----|----------|
| id | UUID | Первичный ключ |
| tm_id | UUID | Ссылка на ТМ |
| engineer_id | UUID | Ссылка на инженера |
| created_at | TIMESTAMP | Дата создания |

**Таблица object_equipment** — привязка оборудования к объектам (адресам). Структура описана в разделе 6.6. Используется для автозаполнения чек-листа при создании визита.

**Таблица equipment_proposals** — предложения инженеров по добавлению оборудования в привязку объекта. Структура описана в разделе 6.7. Статусы: `pending`, `approved`, `rejected`.
