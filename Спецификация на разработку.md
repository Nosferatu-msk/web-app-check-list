# СПЕЦИФИКАЦИЯ НА РАЗРАБОТКУ
## Web-приложение «Цифровой чек-лист инженера по техническому обслуживанию оборудования»

**Версия:** 10.0
**Дата:** 20.07.2026
**Статус:** Готов к разработке

---

## 1. ОБЗОР ПРОЕКТА

### 1.1. Наименование
Web-приложение «Цифровой чек-лист инженера по техническому обслуживанию оборудования» (далее — Приложение).

### 1.2. Назначение
Автоматизация деятельности инженера по техническому обслуживанию оборудования:
- Создание чек-листа работ на объекте
- Фиксация контролируемых параметров оборудования
- Фотофиксация состояния оборудования «до» и «после»
- Формирование PDF-отчёта с маркированными фотографиями
- Отправка отчёта заказчику по email

### 1.3. Ключевые ограничения
- **Офлайн-работа** — обязательна (подвалы, серверные, тепловые узлы без связи)
- **Мобильный интерфейс** — основной сценарий инженера (смартфон/планшет)
- **Соответствие 152-ФЗ** — предупреждения о персональных данных при фотофиксации
- **Все тексты интерфейса** — на русском языке

- Максимум 20 избранных объектов на пользователя
- Дубликаты запрещены (UNIQUE constraint)
- Инженер может изменять только свою специализацию; ТМ/Админ — специализацию подчинённых инженеров

### 4.8. Маппинг специализаций

| Специализация | Поле | Коды оборудования |
|:-------------:|------|-------------------|
| ВиК | `specialization_vik` | vent (деактивирован), vrv_vn, vrv_nar, mssvn, mssnar, splitvn, splitnar, teplozavesa, pritochnaya, pritochno-vytyzhnaya, vytyzhnaya |

## 6. БИЗНЕС-ЛОГИКА

### 6.1. Создание визита

1. Инженер выбирает адрес из справочника (поиск с автодополнением)
2. Поля «Дата начала» и «Время начала» заполняются автоматически (текущие дата/время)
3. Поле «Сезон» вычисляется автоматически по дате:
   - Апрель (1+) – Октябрь (31) → «Лето»
   - Ноябрь (1) – Март (31) → «Зима»
4. При изменении даты — пересчёт сезона с подтверждением пользователя

### 6.2. Управление задачами

1. Кнопка «+ Добавить оборудование» → модальное окно:
   - Вид оборудования (dropdown из справочника)
   - Тип помещения (dropdown из справочника)
   - Местоположение / Назначение (текстовое поле, необязательно)
2. Новая задача получает статус «Не начато»
3. Клик на задачу → переход на страницу параметров (4.x)
4. Статус задачи меняется на «В работе» при первом редактировании
5. Статус «Выполнено» — после сохранения параметров с заключением

### 6.3. Условная обязательность

- Поле «Дополнительные рекомендации» обязательно ТОЛЬКО при:
  - Заключение = «Исправно, есть замечания» ИЛИ
  - Заключение = «Неисправно»
- Валидация при сохранении задачи

### 6.4. Фотофиксация

1. Количество фото определяется типом оборудования:
   - `photos_required = 2` (СС, МСС, VRV, вентиляция): фото ДО + фото ПОСЛЕ
   - `photos_required = 1` (РЩ, счётчики, сети): только фото ДО
2. Для оборудования с 1 фото — поле «Фото ПОСЛЕ» скрыто
3. Предупреждение 152-ФЗ — при первом фото в помещениях с типами:
   - Клиентский зал, Зона самообслуживания, Касса, Санузел, Крыльцо
4. Фото сохраняются в файловое хранилище, метаданные — в БД

### 6.5. Завершение визита

1. Кнопка «Завершить визит» активна при наличии ≥1 выполненной задачи
2. При завершении:
   - Заполняется `time_end`
   - Статус визита → `completed`
   - Генерируется PDF-отчёт
   - Формируется ZIP-архив (PDF + фото)
3. Переход на страницу отправки отчёта (Страница 6)

### 6.6. Маркировка фотографий

**Шаблон имени файла:**
```
{№}_{equipment_code}_{room_code}_{moment}.jpg
```

- `{№}` — порядковый номер задачи в визите (01, 02, 03...)
- `{equipment_code}` — код типа оборудования из справочника
- `{room_code}` — код типа помещения из справочника
- `{moment}` — `before` или `after`

**Примеры:**
- `01_rsch_server_before.jpg`
- `02_splitvn_server_before.jpg`
- `02_splitvn_server_after.jpg`

### 6.7. Формирование PDF-отчёта

**Структура документа** (согласно `Шаблон акта.md`):

1. Заголовок: «Акты выполненных работ»
2. Общий блок:
   - Адрес объекта
   - Дата проведения
   - Время начала / окончания
   - Сезон
   - Инженер (ФИО из профиля)
3. Для каждой задачи:
   - Номер и вид оборудования + местоположение
   - Местоположение (тип помещения)
   - Контролируемые параметры (список)
   - Фотофиксация (встроенные изображения)
   - Заключение
   - Рекомендации (выбранные + дополнительные)
4. Подпись инженера

**Технические требования:**
- Формат: PDF/A-1b
- Страница: A4
- Шрифт: Arial / Roboto, 11pt
- Фото: 150 DPI, встроенные в PDF
- Имя файла: `OTCHET_{street}_{house}_{date}.pdf`

### 6.8. Формирование ZIP-архива

```
OTCHET_{street}_{house}_{date}.zip
├── OTCHET_{street}_{house}_{date}.pdf
└── Photos/
    ├── 01_rsch_server_before.jpg
    ├── 02_splitvn_server_before.jpg
    ├── 02_splitvn_server_after.jpg
    └── ...
```

### 6.9. Отправка email

- **Протокол:** SMTP / SMTPS
- **Кому:** `customer_email` из справочника адресов
- **Тема:** `Акт выполненных работ: {full_address} от {dd.mm.yyyy}`
- **Тело письма:** Шаблон с краткой сводкой (адрес, дата, кол-во задач)
- **Вложение:** ZIP-архив (до 25 МБ)
- **Резерв:** При недоступности email — кнопка «Скачать»

### 6.10. Офлайн-режим

1. **Service Worker** кэширует:
   - Статические ресурсы (HTML, CSS, JS, иконки)
   - API-ответы справочников
2. **IndexedDB** хранит:
   - Текущие визиты и задачи
   - Очередь изменений (pending mutations)
   - Загруженные фото (до синхронизации)
3. **Автосохранение:** каждые 30 секунд (хук `useAutoSave`)
4. **Автосинхронизация:** при появлении сети — отправка накопленных данных
5. **Предупреждение:** при закрытии вкладки с несохранёнными данными (`beforeunload`)

### 6.11. Сброс задачи

- Кнопка «Сбросить задачу» очищает ТОЛЬКО текущую задачу
- Сбрасываются: параметры, рекомендации, фото, заключение
- Статус задачи → «Не начато»
- Остальные задачи визита не затрагиваются

### 6.12. Офлайн-режим

**Локальное хранилище (Dexie / IndexedDB):**
- Таблицы: visits, tasks, photos, syncQueue, cachedRefs
- Каждая запись имеет флаг `dirty` (требует синхронизации)
- Фото хранятся как Blob

**Синхронизация:**
1. При восстановлении связи — `processSyncQueue()` обрабатывает очередь
2. Каждая операция воспроизводится на сервере (create → POST, update → PUT, delete → DELETE)
3. При успехе — запись удаляется из очереди, `dirty = false`
4. При ошибке — retry до 5 раз, затем исключение из очереди
5. После push — `pullVisitsFromServer()` обновляет данные с сервера

**Стратегии кэширования (Workbox):**
| Ресурс | Стратегия | TTL |
|--------|-----------|-----|
| Изображения | CacheFirst | 30 дней |
| /api/refs/* | StaleWhileRevalidate | 1 день |
| /api/* (прочее) | NetworkFirst (timeout 3с) | 7 дней |

### 14.1. Основной сценарий (Happy Path)

1. Инженер входит в систему
2. Создаёт визит, выбирает адрес

8. Скачивает ZIP-архив (PDF + маркированные фото) на устройство

### 14.2. Офлайн-сценарий

1. Инженер создаёт визит без интернета
2. Заполняет задачи и делает фото
3. Данные сохраняются в IndexedDB
4. При появлении сети — автосинхронизация
5. Отчёт формируется и отправляется

### 14.3. Административный сценарий

1. Администратор добавляет новый адрес
2. Привязывает email заказчика
3. Добавляет новую рекомендацию
4. Создаёт учётную запись инженера

### 14.4. Сценарий ТМ

1. Вход как ТМ (tm@example.com / tm12345)
2. Просмотр дашборда с визитами инженеров
3. Фильтрация по конкретному инженеру
4. Просмотр деталей визита инженера
5. Переназначение визита другому инженеру
6. Формирование и отправка отчёта

## 15. ПРИНЯТЫЕ РЕШЕНИЯ (уточнения к ТЗ)

> Ниже зафиксированы решения по вопросам, отсутствующим в исходном техническом задании.

### 15.1. Технологический стек ✅
- **Frontend:** React 18+ / TypeScript, Zustand, Ant Design Mobile / Vant, Workbox
- **Backend:** Node.js / Express, Prisma ORM, Nodemailer
- **Генерация ZIP:** JSZip (на клиенте)

### 15.2. Источник данных инженера ✅
ФИО инженера **вводятся вручную** при создании визита (поле в форме визита). В последующих визитах поле автозаполняется последним использованным значением.

### 15.3. Время окончания визита ✅
`time_end` заполняется **автоматически** при нажатии «Завершить визит» (текущее серверное время).

### 15.4. История визитов ✅
Инженер видит **список своих визитов** с возможностью:
- Просмотра деталей визита
- **Повторного скачивания** PDF + ZIP отчёта
- Фильтрации по дате и статусу

### 15.5. Удаление визитов ✅
Инженер может удалить визит в **любом статусе** (в работе, завершён, отправлен). При удалении удаляются все связанные задачи и фото.

### 15.6. Восстановление пароля ✅
Механизм **сброса по email**: пользователь запрашивает сброс → получает письмо со ссылкой → устанавливает новый пароль. Ссылка действительна 1 час.

### 15.7. SMTP-конфигурация ✅
На этапе разработки — **Ethereal Email** (тестовый SMTP). В production — будет настроен реальный SMTP-сервер (конфигурация через .env).

### 15.8. Хранилище фотографий ✅
**Локальная файловая система** сервера (директория `uploads/`). В дальнейшем планируется миграция на S3-совместимое хранилище. Максимальный размер фото: 10 МБ, максимальное разрешение: 4032×3024 (авто-ресайз при загрузке).

### 15.9. Нумерация задач в фото ✅
Порядковый номер `{№}` — номер задачи в рамках визита. При удалении задачи **номера пересчитываются** для оставшихся задач.

### 15.10. Редактирование завершённого визита ✅
Завершённый визит **можно редактировать**. При повторном сохранении PDF перегенерируется автоматически.

### 15.11. Конфликты синхронизации ✅
Стратегия **«последняя запись побеждает»** (last-write-wins) с timestamp. При конфликте справочников — приоритет у серверной версии.

### 15.12. Уведомления ✅
Push-уведомления **не требуются**. Статусы отображаются в интерфейсе.

### 15.13. Логирование и аудит ✅
**Полный журнал аудита** — таблица `audit_log`:
- id, user_id, action, entity_type, entity_id, old_value, new_value, created_at
- Логируются: вход/выход, создание/редактирование/удаление визитов, задач, справочников

### 15.14. Резервное копирование ✅
Ежедневный бэкап PostgreSQL (pg_dump) + архив директории uploads. Срок хранения: 30 дней.

### 15.15. Домен и SSL ✅
- **Домен:** checkonout.ru (привязан к IP 31.128.38.54)
- **SSL:** Let's Encrypt, сертификат получен через Certbot, автообновление
- **HTTPS:** Nginx слушает 443, HTTP (80) → редирект 301 на HTTPS
- **Хостинг:** Beget VPS, 14 ГБ диск, 1.9 ГБ RAM + 2 ГБ swap

### 15.16. Отправка по email — исключена из UI ✅
Функционал отправки отчёта по email **удалён из пользовательского интерфейса** по решению заказчика. Вместо него реализовано **скачивание ZIP-архива** (PDF + фото) на устройство инженера. Серверный эндпоинт `POST /api/reports/:id/report/send` сохранён для будущего использования.

### 15.17. Формат PDF ✅
Генерируется **стандартный PDF** через Puppeteer (HTML → PDF). Формат PDF/A-1b (из ТЗ) не реализован — Puppeteer не поддерживает PDF/A из коробки. Для долгосрочного архивирования достаточно обычного PDF с встроенными шрифтами.

### 15.18. Источник фото — камера и галерея ✅
На экране фотофиксации доступны **две кнопки**: «Камера» (с `capture="environment"`) и «Галерея» (без `capture`). Пользователь сам выбирает способ получения изображения.

### 15.19. Предупреждение 152-ФЗ — при каждом входе ✅
Предупреждение показывается **каждый раз** при входе на экран фотофиксации в клиентской зоне (без ограничения по sessionStorage).

### 15.20. Автосохранение параметров при переходе к фото ✅
При нажатии кнопки «Фото» в TaskPage параметры **автоматически сохраняются** на сервер перед переходом к PhotoPage. Это предотвращает потерю данных, если пользователь не нажал «Сохранить» явно.

### 15.21. Тема оформления — стандартная светлая ✅
Используется стандартная светлая тема Ant Design (без `darkAlgorithm`). Оптимальна для чтения на мобильных устройствах при любом освещении.

### 15.22. Серверное обновление задач — частичное ✅
Серверный эндпоинт `PUT /tasks/:id` обновляет **только переданные поля** (проверка `!== undefined`). Это предотвращает случайное обнуление `conclusion` и `parameters` при обновлении статуса из PhotoPage.

### 15.23. Адаптивный дизайн админ-панели ✅
Админ-панель адаптирована под все размеры экранов:
- **Мобильные (< 768px):** навигация через Drawer (гамбургер-меню), таблицы с горизонтальной прокруткой, формы на всю ширину, модальные окна адаптированы
- **Планшеты (768–1199px):** боковое меню (Sider), адаптивный контент
- **Десктоп (≥ 1200px):** полноценный Sider, контент с максимальной шириной

Поддерживаемые устройства: iPhone 13–17, Samsung Galaxy, Xiaomi, POCO, OnePlus, iPad, ПК (1920×1080+).

### 15.24. Роль «Территориальный менеджер» (ТМ) ✅
- ТМ видит визиты только своих инженеров (через таблицу tm_engineers)
- ТМ может создавать визиты с назначением на инженера (статус planned)
- ТМ может переназначать визиты между своими инженерами
- ТМ может редактировать, отправлять отчёты, удалять визиты своих инженеров
- ТМ не имеет доступа к справочникам (админ-панели)
- Soft delete: визит помечается is_deleted=true, скрывается из списка

### 15.25. Новые статусы визитов ✅
- planned — ТМ назначил визит инженеру
- not_started — Инженер создал визит, работы не начаты
- sent_by_engineer — Отправлено инженером
- sent_by_tm — Отправлено ТМ
- corrected_by_tm — ТМ внёс корректировки в завершённый визит

### 15.26. Офлайн-режим
- Dexie.js (IndexedDB) для локального хранения визитов, задач, фото
- Очередь мутаций (syncQueue) с автоматической синхронизацией при reconnect
- Workbox NetworkFirst для API (timeout 3с) — кэширует ответы для офлайн-чтения
- Фото хранятся как Blob в IndexedDB до загрузки на сервер

### 15.27. Массовый импорт CSV
- 7 типов импорта: адреса, оборудование, помещения, рекомендации, пользователи, привязки ТМ
- Автоопределение разделителя (`,` / `;`)
- Обработка дубликатов (пропуск), отчёты об ошибках
- Временный пароль `Welcome2026!` для импортированных пользователей

### 15.28. Привязка «Объект-Оборудование»
- Таблица object_equipment: конкретные единицы оборудования на объекте
- Поле object_code в addresses (формат RU/77/001)
- Поле `room_type_code` — **опционально** (nullable). Оборудование может быть привязано к объекту без указания помещения
- При импорте object-equipment (CSV/XLSX) поле room_type не обязательно — пустое значение → `room_type_code = NULL`
- При создании визита таблица работ **пустая** — оборудование добавляется через модальное окно
- Модальное окно «Добавление оборудования» имеет две вкладки:
  - «Связанное оборудование» — чекбокс-лист из object_equipment (отфильтрован по специализации, исключает уже добавленные через `exclude_visit_id`)
  - «Добавить новое» — форма для добавления оборудования вручную (не из привязки)
- Для оборудования без помещения отображается метка «⚠️ Помещение не указано»
- При выборе такого оборудования появляется inline Select для выбора помещения из справочника
- Кнопка «Добавить» неактивна, пока для всех отмеченных единиц без помещения не выбран тип
- При добавлении: вызов `PATCH /api/refs/object-equipment/:id/room` сохраняет помещение в `object_equipment` (room_confirmed_at, room_confirmed_by)
- Поле `object_equipment_id` в таблице `tasks` связывает задачу с конкретной записью привязки (для фильтрации дубликатов)
- При повторном открытии модалки уже добавленные единицы скрыты из списка
- Если всё оборудование добавлено — вкладка показывает «Всё оборудование уже добавлено в визит»
- Для нового (несохранённого) визита — автосохранение черновика при первом нажатии «Добавить оборудование»
- Поддержка импорта из Excel (.xlsx) через библиотеку exceljs

### 15.29. Soft delete адресов ✅
- Поле `is_deleted` в таблице `addresses` (default: false)
- `DELETE /api/admin/addresses/:id` устанавливает `is_deleted = true` вместо физического удаления
- Все списки и поиск адресов фильтруют `is_deleted = false` по умолчанию
- Параметр `?include_deleted=true` показывает удалённые адреса
- Исторические визиты и связанные записи сохраняются

### 15.30. Профиль ТМ ✅
- ТМ видит **закреплённые объекты** (из `tm_objects`) вместо избранных
- `GET /api/profile/objects` — возвращает список объектов с индикатором неисправностей
- Список объектов содержит **клиентскую пагинацию** (по 10/20/50 на странице) с отображением «1–10 из N» и счётчиком общего количества
- ТМ может **создавать инженеров** с автоматической привязкой: `POST /api/profile/engineers`
- Новый инженер получает временный пароль `Welcome2026!` на email

### 15.31. Аутентификация — специализация ✅
- `/auth/login` и `/auth/me` возвращают поля `specializationVik`, `specializationIszh`, `mustChangePassword`
- `trust proxy = 1` для корректной работы rate-limiter за nginx

### 15.32. Избранное — совместимость форматов ✅
- `POST /profile/favorites` принимает `{ addressId }` (UUID) или `{ objectCode }` (строка)
- Сервер резолвит `objectCode` из `addresses` при передаче `addressId`
- `DELETE /profile/favorites/*` — wildcard-маршрут для object_code со слешами (RU/77/001)
